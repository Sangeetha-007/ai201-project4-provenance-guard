# Provenance Guard

A Flask API that detects AI-generated text using two complementary signals — an LLM-based classifier and stylometric heuristics — and returns a transparency label with a confidence score.

---

## How to Run

```bash
# Install dependencies
pip install -r requirements.txt

# Add your Groq API key to .env
echo "GROQ_API_KEY=your_key_here" > .env

# Start the server
python -m flask run -p 5000 --debug
```

---

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/submit` | Submit text for classification |
| GET | `/log` | Return the 50 most recent audit log entries |
| POST | `/appeal` | Submit an appeal for a `likely_ai` result |

### POST `/submit`

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "Your text here.", "creator_id": "user-1"}' \
  | python -m json.tool
```

**Response:**
```json
{
  "content_id": "3f7a2b1e-...",
  "attribution": "likely_ai",
  "confidence": 0.7867,
  "label": "This content was likely generated by an AI. Confidence: 79%. If you believe this is incorrect, you may submit an appeal."
}
```

### GET `/log`

```bash
curl -s http://localhost:5000/log | python -m json.tool
```

Returns structured entries with `llm_score`, `style_score`, `confidence`, and `attribution` for every submission.

### POST `/appeal`

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "<id>", "explanation": "I wrote this myself."}' \
  | python -m json.tool
```

Only available for submissions with `attribution: likely_ai`. Updates the submission status to `under_review` and logs the appeal.

## Detection Signals

### Signal 1 — LLM Classifier (Groq)

The primary signal asks `llama-3.3-70b-versatile` to estimate the probability that a piece of text was AI-generated, returning a float in [0, 1]. This captures semantic and stylistic coherence holistically — things like formulaic phrasing, hedging constructions ("it is important to note that"), and unnatural uniformity of tone that are hard to quantify but that a language model recognises immediately.

**Why this signal:** Rule-based heuristics miss AI text that is stylistically unusual or has been lightly edited. An LLM classifier generalises better across writing styles because it's been trained on both human and AI-generated text at scale.

**What I'd change for production:** The prompt returns a raw probability estimate from the model, which isn't calibrated. In production I'd use a fine-tuned binary classifier with Platt scaling or isotonic regression to produce calibrated probabilities, and I'd track its precision/recall on a held-out labelled dataset rather than relying on intuitive testing.

### Signal 2 — Stylometric Heuristics

A pure-Python composite of three structural features, each normalised to [0, 1] where 1 = AI-like:

- **Sentence-length variance** — AI text tends to have more uniform sentence lengths. Low variance → score near 1: `1 / (1 + variance(sentence_word_counts))`
- **Type-token ratio (TTR)** — AI text reuses vocabulary less than humans in short passages but uses a narrower semantic range over longer ones. Low TTR → score near 1: `1 - (unique_words / total_words)`
- **Punctuation-count variation** — AI text applies punctuation consistently across sentences. Low coefficient of variation → score near 1: `1 / (1 + stdev / mean)`

`style_score = mean(slv_score, ttr_score, punct_score)`

**Why this signal:** It is independent of the LLM classifier — it measures structural regularity rather than semantic content. The two signals catch different failure modes: the LLM may miss stylistically unusual AI text; the heuristics may miss semantically coherent but structurally unremarkable AI text.

**What I'd change for production:** The current normalisation maps almost all text — human or AI — into the 0.15–0.30 range, which means it contributes little discrimination. I'd replace the hand-crafted normalisation with a logistic regression trained on a labelled corpus so the feature weights and scaling reflect real distributional differences rather than intuition.

## Confidence Scoring

```
confidence = 0.8 × llm_score + 0.2 × style_score
```

**Why this weighting:** The original spec used 0.6/0.4. After testing, stylometric scores clustered tightly around 0.15–0.26 for all text types regardless of whether the text was human or AI. The LLM signal, by contrast, ranged from 0.21 (clearly human) to 0.92 (clearly AI). Giving equal weight to a weak signal and a strong one would pull confidence toward the centre and make the `likely_ai` bucket unreachable. The 0.8/0.2 split reflects where the actual discrimination power lives.

**Thresholds:**
- `≥ 0.70` → `likely_ai`
- `0.40 – 0.69` → `uncertain`
- `< 0.40` → `likely_human`

The threshold was also adjusted from the spec's 0.75 to 0.70 after seeing that AI text consistently scored in the 0.73–0.79 range — 0.75 put the bucket boundary in the middle of the AI cluster rather than above it.

### Example Submissions

**High-confidence AI** (formal, templated paragraph):
> "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment."

| llm_score | style_score | confidence | attribution |
|-----------|-------------|------------|-------------|
| 0.92 | 0.2535 | 0.7867 | `likely_ai` |

**Lower-confidence case** (factual but neutral — the water cycle):
> "The water cycle describes the continuous movement of water within Earth and its atmosphere. Evaporation occurs when the sun heats surface water, which then rises as vapor. It condenses into clouds, and precipitation returns it to the surface."

| llm_score | style_score | confidence | attribution |
|-----------|-------------|------------|-------------|
| 0.86 | 0.22 | 0.6988 | `uncertain` |

The 9-point gap in confidence (0.79 vs. 0.70) reflects that the factual paragraph is structurally similar to AI output but the LLM recognises it as the kind of neutral prose that appears in textbooks and encyclopaedias — not confidently machine-generated.

## Transparency Labels

Each submission returns one of three label strings based on the confidence score. The exact text displayed:

**`likely_ai` (confidence ≥ 0.70):**
> "This content was likely generated by an AI. Confidence: {score}%. If you believe this is incorrect, you may submit an appeal."

**`uncertain` (confidence 0.40–0.69):**
> "Our system could not confidently determine whether this content is human- or AI-written. Confidence: {score}%. No action has been taken."

**`likely_human` (confidence < 0.40):**
> "This content appears to be human-written. Confidence: {score}%. No action has been taken."

The appeal prompt appears only on the `likely_ai` label — it is the only outcome that carries a consequence for the submitter. The `{score}` is rendered as a percentage rounded to the nearest integer (e.g. `79%`).

## Appeals Workflow

Any submitter who received a `likely_ai` label can POST to `/appeal` with their `content_id` and an optional explanation. The system:

1. Validates that the submission exists and has `attribution: likely_ai` (returns 403 otherwise)
2. Updates the original audit log entry status from `classified` to `under_review`
3. Appends a separate `appeal` event to the log with the explanation and timestamp
4. Returns `{"status": "under_review"}`

A human reviewer opening the audit log would see the original classification entry (with `llm_score`, `style_score`, `confidence`) and the follow-on appeal event (with the submitter's explanation), giving them enough context to make a decision.

## Known Limitations

**Factual reference prose will produce false positives.** Wikipedia-style text, textbook explanations, and technical documentation are structurally indistinguishable from AI output to both signals. The LLM classifier scores this kind of text high (0.80+) because it was trained on vast amounts of encyclopaedic text that now forms part of many AI systems' training data. The stylometric signal also scores it high because reference prose is deliberately uniform. A student who writes in a clear, factual style to explain a concept will likely be flagged as `uncertain` or `likely_ai` even if every word is their own.

This is not a data problem — it is a structural property of the signals: both are measuring uniformity and formulaicity, which are features of good technical writing as well as AI writing.

**Short texts are unreliable.** The stylometric signal requires at least two sentences and degrades further below five. The sentence-length variance and punctuation variation calculations become noisy with few data points, causing the `style_score` to fall back to a neutral 0.5 and the LLM score to dominate entirely. A one- or two-sentence submission from either a human or an AI will receive a poorly-calibrated result.

## Spec Reflection

**One way the spec helped:** Writing out the three transparency label variants in `planning.md` before touching any code forced a concrete decision about what "uncertain" means to a non-technical reader. The label text ("Our system could not confidently determine...") was written in the spec, not in the implementation — which meant the implementation had a clear target rather than a vague notion of "show a message."

**One way the implementation diverged:** The spec set signal weights at 0.6/0.4 (LLM/stylometric) and a `likely_ai` threshold at 0.75. Both had to change after testing. The stylometric signal consistently scored in the 0.15–0.26 range for all text types, meaning 0.4 weight on it was dragging every confidence score toward a narrow band around 0.42–0.55. The weights shifted to 0.8/0.2 and the threshold dropped to 0.70 to match the actual score distribution. The spec was right that the signals are complementary — it was wrong about how much discrimination power the stylometric signal would contribute at the lengths of text being tested.

## AI Usage

**Instance 1 — Stylometric scoring function**

I directed Claude to generate a Python function that measures sentence-length variance, type-token ratio, and punctuation-count variation, normalising each to [0, 1] where 1 = AI-like. It produced a working implementation using the `statistics` and `re` standard library modules. I overrode the direction of the punctuation signal — the initial draft treated high punctuation density as AI-like, but AI text is not necessarily more punctuated than human text; what differs is the *uniformity* of punctuation across sentences. I revised the signal to use the coefficient of variation of per-sentence punctuation counts rather than the raw density.

**Instance 2 — Confidence weights and threshold**

I asked Claude to explain why the `likely_ai` bucket was unreachable during testing. It diagnosed that with weights 0.6/0.4 and stylometric scores clustering at 0.25, the maximum achievable confidence with a perfect LLM score was 0.70 — below the 0.75 threshold. It proposed two options: reweight to 0.8/0.2 or lower the threshold. I made both changes — the reweight because the data showed the LLM was the stronger signal, and the threshold adjustment because 0.75 was landing in the middle of the AI text cluster rather than above it. The decision to do both (rather than just one) was mine; Claude's diagnosis was correct but the judgment call on the fix required seeing the actual score distributions from testing.

**Instance 3 — Log endpoint and audit structure**

I directed Claude to set up an append-only JSON Lines audit log with a `GET /log` endpoint. It produced the `write_audit` / `get_log` helper functions and the endpoint. I revised the appeal flow to write a *separate* appeal event rather than mutating the original classification entry in place — the first draft only updated the status field, which would have destroyed the original `classified` state. The final implementation rewrites the original entry's status field AND appends a distinct `appeal` event, preserving the full history for a human reviewer.
